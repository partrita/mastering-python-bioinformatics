실전 생명정보학을 위한 파이썬 마스터하기 (Mastering Python for Bioinformatics)

오늘날의 생명과학자들에게 생명정보학 기술 교육은 시급한 과제입니다. 너무나 많은 생명정보학 프로그램이 제대로 작성되지 않았거나 유지보수가 거의 이루어지지 않고 있습니다. 이는 주로 기초 프로그래밍 기술을 제대로 배우지 못한 학생이나 연구자들에 의해 작성되었기 때문입니다. 이 실전 가이드는 포닥(postdoc) 생명정보학 전문가와 학생들이 파이썬의 핵심 기능을 활용하여 생물학적 문제를 해결하는 동시에, 문서화되고 테스트 가능하며 재현 가능한 소프트웨어를 만드는 방법을 보여줍니다.

서문 (Preface)

프로그래밍은 능력의 배가(force multiplier)를 가져옵니다. 우리는 컴퓨터 프로그램을 작성하여 지루한 수동 작업에서 벗어나고 연구 속도를 높일 수 있습니다. 어떤 언어로든 프로그래밍을 하면 생산성이 향상되겠지만, 각 언어는 코딩 과정을 개선하거나 방해하는 서로 다른 학습 곡선과 도구를 가지고 있습니다.

비즈니스 세계에는 세 가지 선택지 중 두 가지만 고를 수 있다는 격언이 있습니다:

  1. 속도 (Fast)
  2. 품질 (Good)
  3. 비용 (Cheap)

프로그래밍 언어의 관점에서 볼 때, 파이썬은 이 세 가지 사이에서 절묘한 균형점을 찾았습니다. 배우기 쉽고 아이디어의 프로토타입을 빠르게 작성할 수 있기 때문에 ‘빠릅니다(Fast)’. 저는 어떤 프로그램을 작성할 때든 거의 항상 파이썬을 첫 번째 언어로 선택합니다. 또한 파이썬 프로그램은 내 노트북이나 작은 AWS 인스턴스와 같은 일반적인 하드웨어에서도 충분히 잘 돌아가기 때문에 ‘저렴합니다(Cheap)’. 하지만 파이썬 자체의 규범이 상당히 느슨하기 때문에 파이썬으로 ‘좋은(Good)’ 프로그램을 만드는 것이 반드시 쉬운 일만은 아니라고 생각합니다. 예를 들어, 파이썬은 연산 중에 문자와 숫자가 섞이는 것을 허용하여 프로그램이 예기치 않게 중단될 수 있습니다.

이 책은 다음과 같은 파이썬의 모범 사례와 도구들을 배우고 싶어 하는 예비 생명정보학 프로그래머들을 위해 쓰였습니다:

  • 타입 힌트 (Type Hints): 파이썬 3.6부터 변수가 숫자나 리스트와 같은 특정 타입이어야 함을 나타내는 타입 힌트를 추가할 수 있으며, mypy 도구를 사용해 타입이 올바르게 사용되었는지 확인할 수 있습니다.
  • 테스트 프레임워크 (Testing Frameworks): pytest와 같은 도구를 사용하여 코드가 정상 데이터와 비정상 데이터 모두에 대해 예측 가능한 방식으로 반응하는지 테스트할 수 있습니다.
  • 린팅 도구 (Linting Tools): pylint나 flake8과 같은 도구를 사용하여 잠재적인 오류와 스타일 문제를 찾아내어 프로그램을 더 이해하기 쉽게 만들 수 있습니다.
  • argparse 모듈: 프로그램의 인자를 문서화하고 검증할 수 있습니다.
  • 생태계 활용: 바이오파이썬(Biopython)과 같은 수백 개의 기존 모듈을 활용하여 프로그램을 짧고 더 안정적으로 만들 수 있습니다.

이러한 도구들을 개별적으로 연습하는 것만으로도 프로그램이 개선되겠지만, 이들을 모두 결합하면 복합적인 방식으로 코드를 향상시킬 수 있습니다. 이 책은 생명정보학 그 자체를 다루는 교과서는 아닙니다. 대신 파이썬이 제공하는 기능들 중 ‘재현 가능한(reproducible)’ 과학 프로그램을 작성하는 데 적합한 기능들에 초점을 맞춥니다. 즉, 동일한 입력이 주어졌을 때 항상 동일한 출력을 생성하는 프로그램을 설계하고 테스트하는 방법을 보여드릴 것입니다. 생명정보학 분야에는 제대로 작성되지 않고 문서화되지 않은 프로그램들이 너무 많습니다. 제 목표는 이러한 추세를 하나씩 바꿔나가는 것입니다.

프로그램 재현성을 위한 기준은 다음과 같습니다:

파라미터화 (Parameters)

모든 프로그램 파라미터는 실행 시 인자로 설정할 수 있어야 합니다. 즉, 프로그램의 동작을 변경하기 위해 소스 코드를 직접 수정해야 하는 하드코딩된 값이 없어야 함을 의미합니다.

문서화 (Documentation)

프로그램은 --help 인자에 반응하여 파라미터와 사용법을 출력해야 합니다.

테스트 (Testing)

코드가 특정 사양을 충족함을 증명하는 테스트 스위트를 실행할 수 있어야 합니다.

대부분의 생명정보학자들은 프로그래밍을 배운 과학자이거나 생물학을 배운 프로그래머들입니다. 어떤 경로로 생명정보학 분야에 들어왔든, 여러분이 올바른 프로그램을 빠르게 작성할 수 있도록 돕는 실용적인 프로그래밍 기법을 보여드리고자 합니다. 먼저 프로그램의 인자를 문서화하고 검증하는 방법부터 시작하여, 프로그램이 의도한 대로 작동하는지 확인하기 위해 테스트를 작성하고 실행하는 방법을 살펴보겠습니다.

예를 들어, 1장에서는 DNA 문자열에서 4중 뉴클레오티드(tetranucleotide) 빈도를 보고하는 방법을 보여줍니다. 매우 간단해 보이죠? 사소한 아이디어일 수 있지만, 저는 이 프로그램을 어떻게 구조화하고, 문서화하며, 테스트하는지 보여주는 데 약 40페이지를 할애할 것입니다. 프로그램의 여러 버전을 작성하고 테스트하면서 파이썬의 데이터 구조, 문법, 모듈 및 도구의 다양한 측면을 탐구할 것입니다.

독자 대상 (Who Should Read This?)

프로그래밍이라는 공예(craft)에 관심이 있고, 문서화되고 파라미터를 검증하며, 우아하게 실패하고 안정적으로 작동하는 프로그램을 작성하는 법을 배우고 싶다면 이 책을 읽어야 합니다. 테스트는 여러분의 코드를 이해하고 정확성을 검증하는 데 핵심적인 기술입니다. 제가 작성한 테스트를 활용하는 방법뿐만 아니라 여러분의 프로그램을 위한 테스트를 작성하는 방법도 보여드리겠습니다.

이 책을 최대한 활용하려면 이미 파이썬에 대한 어느 정도의 이해가 있어야 합니다. 저는 Tiny Python Projects (Manning, 2020)에서 가르친 기술들, 즉 문자열, 리스트, 튜플, 딕셔너리, 세트, 네임드 튜플과 같은 파이썬 데이터 구조를 활용하는 방법을 바탕으로 내용을 전개할 것입니다. 파이썬 전문가일 필요는 없지만, 타입, 정규 표현식, 고차 함수(higher-order functions), 테스트 및 pylint, flake8, yapf, pytest와 같은 도구들을 사용하여 스타일, 문법, 정확성을 확인하는 등의 고급 개념들을 이해하도록 독려할 것입니다. 특히 이 책의 모든 코드에는 일관되게 타입 어노테이션(type annotations)을 사용하며, mypy 도구를 통해 타입의 올바른 사용을 보장할 것입니다.

프로그래밍 스타일: OOP와 예외 처리를 피하는 이유 (Programming Style: Why I Avoid OOP and Exceptions)

저는 객체 지향 프로그래밍(OOP, Object-Oriented Programming)을 피하는 경향이 있습니다. OOP가 무엇인지 모르셔도 괜찮습니다. 파이썬(Python) 자체는 객체 지향 언어이며, 문자열부터 세트에 이르기까지 거의 모든 요소가 기술적으로는 내부 상태와 메서드(methods)를 가진 객체입니다. 여러분은 OOP가 무엇인지 감을 잡을 수 있을 정도로 충분히 많은 객체를 접하게 되겠지만, 제가 제시하는 프로그램들은 대부분 아이디어를 표현하기 위해 객체를 사용하는 것을 피할 것입니다.

물론 1장에서는 클래스(class)를 사용하여 복잡한 데이터 구조를 표현하는 방법을 보여줍니다. 클래스를 사용하면 타입 힌트(type annotations)를 포함한 데이터 구조를 정의할 수 있어, 데이터 타입을 올바르게 사용하고 있는지 확인할 수 있습니다. OOP를 조금 이해하는 것이 도움이 되긴 합니다. 예를 들어, 클래스는 객체의 속성을 정의하고 부모 클래스로부터 속성을 상속받을 수 있지만, 이것이 제가 파이썬에서 OOP를 사용하는 목적과 한계의 전부입니다. 지금 당장 완전히 이해되지 않더라도 걱정하지 마세요. 직접 코드를 보면 이해하게 될 것입니다.

객체 지향 코드 대신, 저는 거의 전적으로 함수로 구성된 프로그램을 보여드릴 것입니다. 이러한 함수들은 전달받은 값에 대해서만 동작한다는 점에서 ‘순수(pure)’합니다. 즉, 순수 함수는 전역 변수와 같이 숨겨진 가변 상태에 의존하지 않으며, 동일한 인자가 주어지면 항상 동일한 값을 반환합니다. 또한 모든 함수에는 예측 가능하게 동작하는지 확인할 수 있는 관련 테스트가 포함될 것입니다. 제 개인적인 의견으로는 이러한 방식이 OOP를 사용하여 작성된 솔루션보다 프로그램이 더 짧고 투명하며 테스트하기 쉽다고 생각합니다. 여러분은 제 의견에 동의하지 않을 수도 있고, 테스트를 통과하기만 한다면 자신이 선호하는 프로그래밍 스타일로 솔루션을 작성해도 괜찮습니다. 파이썬의 ’함수형 프로그래밍 HOWTO’ 문서는 파이썬이 왜 함수형 프로그래밍(FP, Functional Programming)에 적합한지에 대해 잘 설명하고 있습니다.

마지막으로, 이 책의 프로그램들은 예외(exceptions) 사용을 피합니다. 이는 개인이 사용하는 짧은 프로그램에 적합하다고 생각합니다. 프로그램의 흐름을 방해하지 않도록 예외를 관리하는 것은 또 다른 복잡성을 더하며, 프로그램 이해 능력을 떨어뜨린다고 느낍니다. 저는 일반적으로 에러를 반환하는 파이썬 함수를 작성하는 방식에 만족하지 못합니다. 많은 사람이 예외를 발생시키고 try/catch 블록이 이를 처리하도록 합니다. 만약 예외 처리가 정당하다고 느껴진다면, 저는 아예 예외를 잡지 않고 프로그램이 중단되도록 내버려 두는 쪽을 선택하곤 합니다. 이런 면에서 저는 얼랑(Erlang) 언어의 창시자인 조 암스트롱(Joe Armstrong)의 생각을 따르고 있습니다. 그는 “얼랑 방식은 정상적인 경로(happy path)를 작성하는 것이며, 에러 교정 코드로 가득 찬 뒤틀린 작은 통로를 작성하는 것이 아니다”라고 말했습니다.

공개용 프로그램이나 모듈을 작성하기로 했다면 예외와 에러 처리에 대해 훨씬 더 많이 배워야 하겠지만, 이는 이 책의 범위를 벗어납니다.

구조 (Structure)

이 책은 크게 두 부분으로 나뉩니다. 첫 번째 부분은 Rosalind.info 웹사이트에 있는 14가지 프로그래밍 챌린지를 다룹니다. 두 번째 부분은 생명정보학에서 중요하다고 생각되는 다른 패턴이나 개념을 보여주는 더 복잡한 프로그램들을 보여줍니다. 책의 모든 장에서는 여러분이 작성할 코딩 과제를 설명하고, 제대로 작동하는 프로그램을 작성했는지 확인할 수 있는 테스트 스위트를 제공합니다.

파이썬의 선(Zen of Python)은 “어떤 일을 수행하는 명확한 방법은 오직 하나(preferably only one)여야 한다”고 말하지만, 저는 한 가지 문제에 대해 여러 가지 다른 접근 방식을 시도함으로써 많은 것을 배울 수 있다고 믿습니다. 펄(Perl)은 제가 생명정보학으로 들어가는 관문이었고, 펄 커뮤니티의 정신인 “어떤 일을 하는 방법은 여러 가지가 있다(TMTOWTDI, There’s More Than One Way To Do It)”는 여전히 저에게 공감을 불러일으킵니다. 저는 일반적으로 각 장에서 ‘주제와 변주’ 방식을 따라, 파이썬 문법과 데이터 구조의 다양한 측면을 탐구할 수 있는 여러 솔루션을 보여줍니다.

테스트 주도 개발 (Test-Driven Development)

테스트를 수행하는 행위보다 테스트를 설계하는 행위 자체가 알려진 최고의 버그 방지책 중 하나입니다. 유용한 테스트를 만들기 위해 수행해야 하는 사고 과정은 코드가 작성되기 전에 버그를 발견하고 제거할 수 있게 해줍니다. 실제로 테스트 설계 사고는 개념 정립부터 명세화, 설계, 코딩 등에 이르기까지 소프트웨어 제작의 모든 단계에서 버그를 발견하고 제거할 수 있습니다.

—보리스 바이저(Boris Beizer), Software Testing Techniques (Thompson Computer Press)

제 모든 실험의 밑바탕에는 프로그램이 계속해서 올바르게 작동하는지 확인하기 위해 끊임없이 실행할 테스트 스위트가 있습니다. 기회가 될 때마다 저는 켄트 벡(Kent Beck)이 쓴 같은 제목의 책(Addison-Wesley, 2002)에서 설명된 개념인 테스트 주도 개발(TDD, Test-Driven Development)을 가르치려 노력합니다. TDD는 코드를 작성하기 전에 코드에 대한 테스트를 먼저 작성할 것을 권장합니다. 전형적인 사이클은 다음과 같습니다:

  1. 테스트를 추가합니다.
  2. 모든 테스트를 실행하고 새로운 테스트가 실패하는지 확인합니다.
  3. 코드를 작성합니다.
  4. 테스트를 실행합니다.
  5. 코드를 리팩토링합니다.
  6. 반복합니다.

책의 GitHub 저장소에서 여러분이 작성할 각 프로그램에 대한 테스트를 찾을 수 있습니다. 테스트를 실행하고 작성하는 방법을 설명해 드릴 것이며, 이 책을 마칠 때쯤이면 TDD를 사용하는 것이 상식이자 기본적인 미덕임을 믿게 되기를 바랍니다. 테스트를 먼저 생각하는 것이 여러분이 코딩을 이해하고 탐구하는 방식을 바꾸기 시작하기를 바랍니다.

명령줄 사용 및 파이썬 설치 (Using the Command Line and Installing Python)

생명정보학에서의 제 경험은 항상 유닉스(Unix) 명령줄을 중심으로 이루어졌습니다. 제 일상 업무의 상당 부분은 리눅스(Linux) 서버에서 쉘 스크립트, 펄, 파이썬을 사용하여 기존 명령줄 프로그램들을 연결하는 것이었습니다. 노트북에서 프로그램을 작성하고 디버깅할 수도 있지만, 제 도구들을 고성능 컴퓨팅(HPC) 클러스터에 배포하는 경우가 많습니다. 그곳에서는 스케줄러가 프로그램을 비동기적으로 실행하며, 종종 한밤중이나 주말에 저의 감독이나 개입 없이 실행됩니다. 또한 데이터베이스와 웹사이트를 구축하고 서버를 관리하는 제 모든 작업은 전적으로 명령줄에서 이루어집니다. 따라서 생명정보학에서 성공하려면 이 환경을 마스터해야 한다고 굳게 믿습니다.

저는 이 책의 모든 자료를 작성하고 테스트하는 데 매킨토시(Macintosh)를 사용했으며, macOS에는 명령줄로 사용할 수 있는 터미널(Terminal) 앱이 있습니다. 또한 다양한 리눅스 배포판을 사용하여 모든 프로그램을 테스트했으며, GitHub 저장소에는 도커(Docker)와 함께 리눅스 가상 머신을 사용하는 방법에 대한 지침이 포함되어 있습니다. 또한 윈도우 10에서 Linux용 Windows 하위 시스템(WSL) 버전 1의 우분투(Ubuntu) 배포판을 사용하여 모든 프로그램을 테스트했습니다. 윈도우 사용자에게는 진정한 유닉스 명령줄을 가질 수 있는 WSL을 강력히 추천하지만, cmd.exe, PowerShell, Git Bash와 같은 윈도우 쉘도 일부 프로그램에는 충분히 잘 작동할 수 있습니다.

여러분이 프로그램을 작성하고 실행하며 테스트하는 데 도움이 되도록 VS Code, PyCharm, Spyder와 같은 통합 개발 환경(IDE)을 탐구해 보시길 권장합니다. 이러한 도구들은 텍스트 에디터, 도움말 문서, 터미널을 통합하고 있습니다. 저는 터미널에서 vim 에디터를 사용하여 모든 프로그램과 테스트, 심지어 이 책까지 썼지만, 대부분의 사람은 아마도 Sublime, TextMate 또는 Notepad++와 같은 좀 더 현대적인 텍스트 에디터를 사용하는 것을 선호할 것입니다.

저는 파이썬 버전 3.8.6과 3.9.1을 사용하여 모든 예제를 작성하고 테스트했습니다. 일부 예제는 버전 3.6에는 없는 파이썬 문법을 사용하므로, 그 버전은 사용하지 마시길 권장합니다. 파이썬 버전 2.x는 더 이상 지원되지 않으며 사용해서는 안 됩니다. 저는 보통 파이썬 다운로드 페이지에서 최신 버전의 파이썬 3를 내려받지만, 아나콘다(Anaconda) 파이썬 배포판을 사용하여 성공하기도 했습니다. 우분투의 apt나 맥의 brew와 같은 패키지 매니저를 사용하여 최신 버전을 설치할 수도 있고, 소스에서 직접 빌드할 수도 있습니다. 플랫폼과 설치 방법이 무엇이든, 언어가 계속해서 더 나은 방향으로 변화하고 있으므로 가장 최신 버전을 사용하려고 노력하시길 권장합니다.

참고로 저는 주피터 노트북(Jupyter Notebook)이 아닌 명령줄 프로그램으로 프로그램을 제시하기로 했습니다. 여기에는 몇 가지 이유가 있습니다. 데이터 탐색에는 노트북을 좋아하지만, 노트북의 소스 코드는 행 단위 텍스트가 아닌 JSON(JavaScript Object Notation) 형식으로 저장됩니다. 이로 인해 diff와 같은 도구를 사용하여 두 노트북 간의 차이점을 찾기가 매우 어렵습니다. 또한 노트북은 파라미터화(parameterized)할 수 없습니다. 즉, 동작을 변경하기 위해 프로그램 외부에서 인자를 전달할 수 없고 소스 코드 자체를 변경해야 합니다. 이는 프로그램을 유연하지 못하게 만들고 자동화된 테스트를 불가능하게 합니다. 여러분이 노트북을, 특히 파이썬을 실행하는 대화형 방식으로서 탐구해 보시기를 권장합니다, 하지만 저는 명령줄 프로그램을 작성하는 방법에 집중할 것입니다.

코드 및 테스트 가져오기 (Getting the Code and Tests)

모든 코드 및 테스트는 책의 GitHub 저장소에서 확인할 수 있습니다. Git 프로그램(별도 설치 필요)을 사용하여 다음 명령으로 코드를 컴퓨터에 복사할 수 있습니다. 이 명령을 실행하면 저장소 내용이 담긴 biofx_python이라는 새 디렉토리가 생성됩니다.

$ git clone https://github.com/kyclark/biofx_python

IDE를 사용 중이라면 그림 P-1과 같이 해당 인터페이스를 통해 저장소를 클론할 수도 있습니다. 많은 IDE가 프로젝트 관리와 코드 작성을 도와주지만 작동 방식은 저마다 다릅니다. 단순함을 위해 여기서는 대부분의 작업을 명령줄에서 수행하는 방법을 보여드리겠습니다.


그림 P-1. PyCharm 도구를 사용하여 GitHub 저장소를 직접 클론할 수 있습니다.

PyCharm과 같은 일부 도구는 프로젝트 디렉토리 내에 가상 환경(virtual environment)을 자동으로 생성하려고 할 수 있습니다. 이는 컴퓨터의 다른 프로젝트로부터 파이썬 버전과 모듈을 격리하는 방법입니다. 가상 환경 사용 여부는 개인의 취향이며 필수 사항은 아닙니다.

여러분의 계정에 코드 복사본을 만들어 변경 사항을 추적하고 솔루션을 다른 사람들과 공유하고 싶을 수도 있습니다. 이를 ’포킹(forking)’이라고 하는데, 제 코드에서 갈라져 나와 여러분의 프로그램을 저장소에 추가하는 것을 의미합니다.

제 GitHub 저장소를 포크하려면 다음 단계를 따르세요:

  1. GitHub.com에서 계정을 만듭니다.
  2. https://github.com/kyclark/biofx_python으로 이동합니다.
  3. 오른쪽 상단의 ‘Fork’ 버튼(그림 P-2 참조)을 클릭하여 여러분의 계정에 저장소 복사본을 만듭니다.

그림 P-2. 제 GitHub 저장소의 Fork 버튼을 누르면 여러분의 계정에 코드 복사본이 만들어집니다.

이제 저장소에 제 코드의 복사본이 생겼으므로, Git을 사용하여 해당 코드를 여러분의 컴퓨터로 복사할 수 있습니다. YOUR_GITHUB_ID 부분을 실제 GitHub ID로 바꾸는 것을 잊지 마세요:

$ git clone https://github.com/YOUR_GITHUB_ID/biofx_python

여러분이 복사본을 만든 후에 제가 저장소를 업데이트할 수도 있습니다. 이러한 업데이트를 받으려면 제 저장소를 업스트림(upstream) 소스로 설정하도록 Git을 구성해야 합니다. 이를 위해 저장소를 컴퓨터로 클론(clone)한 후 biofx_python 디렉토리로 이동합니다:

$ cd biofx_python

그런 다음 아래 명령을 실행합니다:

$ git remote add upstream https://github.com/kyclark/biofx_python.git

제 저장소에서 여러분의 저장소를 업데이트하고 싶을 때마다 다음 명령을 실행하면 됩니다:

$ git pull upstream main

모듈 설치 (Installing Modules)

여러 파이썬 모듈과 도구를 설치해야 합니다. 저장소 최상위 디렉토리에 requirements.txt 파일을 포함해 두었습니다. 이 파일에는 책의 프로그램을 실행하는 데 필요한 모든 모듈이 나열되어 있습니다. 일부 IDE는 이 파일을 감지하고 설치를 제안할 수도 있으며, 직접 다음 명령을 사용할 수도 있습니다:

$ python3 -m pip install -r requirements.txt

또는 pip3 도구를 사용하세요:

$ pip3 install -r requirements.txt

때때로 pylint가 프로그램의 일부 변수 이름에 대해 경고를 보내거나, mypy가 타입 힌트(type annotations)가 없는 모듈을 임포트할 때 문제를 제기할 수 있습니다. 이러한 에러를 조용히 시키려면 홈 디렉토리에 초기화 파일을 만들어 프로그램의 동작을 사용자 정의할 수 있습니다. 소스 저장소의 루트에는 pylintrc와 mypy.ini라는 파일이 있는데, 이를 다음과 같이 홈 디렉토리로 복사해야 합니다:

$ cp pylintrc ~/.pylintrc
$ cp mypy.ini ~/.mypy.ini

또는 다음 명령으로 새로운 pylintrc를 생성할 수도 있습니다:

$ cd ~
$ pylint --generate-rcfile > .pylintrc

취향에 맞게 이 파일들을 자유롭게 수정해 보세요.

new.py 프로그램 설치 (Installing the new.py Program)

저는 파이썬 프로그램을 만들어주는 new.py라는 파이썬 프로그램을 작성했습니다. 참 메타(meta)하죠? 저는 빈 화면에서 프로그램을 작성하기 시작하는 것이 꽤 어렵다고 생각해서 저 자신을 위해 이 프로그램을 만들었고, 학생들에게도 나누어 주었습니다. new.py 프로그램은 argparse 모듈을 사용하여 명령줄 인자(command-line arguments)를 해석하는 잘 구조화된 새로운 파이썬 프로그램을 생성합니다. 이전 섹션에서 모듈 의존성과 함께 설치되었을 것입니다. 설치되지 않았다면 다음과 같이 pip 모듈을 사용하여 설치할 수 있습니다:

$ python3 -m pip install new-py

이제 new.py를 실행하면 다음과 같은 내용을 볼 수 있을 것입니다:

usage: new.py [-h] [-n NAME] [-e EMAIL] [-p PURPOSE] [-t] [-f] [--version]
program
new.py: error: the following arguments are required: program

각 연습 문제에서는 new.py를 사용하여 새로운 프로그램을 작성할 것을 권장합니다. 예를 들어, 1장에서는 다음과 같이 01_dna 디렉토리에 dna.py라는 프로그램을 만들 것입니다:

$ cd 01_dna/
$ new.py dna.py
Done, see new script "dna.py".

그런 다음 ./dna.py --help를 실행하면 프로그램 사용법에 대한 도움말 문서가 생성되는 것을 볼 수 있습니다. 에디터에서 dna.py 프로그램을 열고 인자를 수정하고 코드를 추가하여 프로그램과 테스트의 요구 사항을 충족시키면 됩니다.

new.py를 사용하는 것이 필수는 아닙니다. 시작하는 데 도움을 주기 위해 제공하는 것일 뿐입니다. 저도 모든 프로그램을 이렇게 시작하지만, 여러분은 다른 방식을 선호할 수도 있습니다. 여러분의 프로그램이 테스트 스위트를 통과하기만 한다면 어떤 방식으로 작성하든 괜찮습니다.

이 책을 쓴 이유 (Why Did I Write This Book?)

리처드 해밍(Richard Hamming)은 벨 연구소(Bell Labs)에서 수학자이자 연구자로 수십 년을 보냈습니다. 그는 모르는 사람들을 찾아가 그들의 연구에 대해 묻는 것으로 유명했습니다. 그러고 나서 그들에게 그들의 분야에서 가장 크고 시급한 미해결 질문이 무엇이라고 생각하는지 물었습니다. 만약 두 질문에 대한 답이 같지 않다면, 그는 이렇게 물었습니다. “그런데 왜 그걸 연구하지 않나요?”

저는 생명정보학의 가장 시급한 문제 중 하나가 많은 소프트웨어가 제대로 작성되지 않았고, 문서화와 테스트가 적절히 이루어지지 않았다는 점이라고 생각합니다. 타입(types), 테스트(tests), 린터(linters), 포매터(formatters)를 사용하는 것이 생각보다 어렵지 않다는 것을 보여드리고 싶습니다. 이러한 도구들을 사용하면 시간이 지남에 따라 새로운 기능을 추가하고 더 나은 소프트웨어를 출시하는 것이 더 쉬워질 것이기 때문입니다. 여러분은 최소한 어느 정도의 정확도 측면에서 여러분의 프로그램이 확실히 올바르다는 자신감을 가질 수 있게 될 것입니다.

이를 위해 소프트웨어 개발의 모범 사례를 보여드리겠습니다. 비록 파이썬(Python)을 매체로 사용하고 있지만, 그 원칙은 C부터 R, 자바스크립트(JavaScript)에 이르기까지 모든 언어에 적용됩니다. 이 책에서 배울 수 있는 가장 중요한 것은 소프트웨어를 개발, 테스트, 문서화, 출시 및 지원하는 기술(craft)이며, 이를 통해 우리 모두가 과학 연구 컴퓨팅을 발전시킬 수 있기를 바랍니다.

생명정보학 분야에서의 제 경력은 방황과 행복한 우연의 산물이었습니다. 대학에서 영문학과 음악을 공부한 후 데이터베이스와 HTML을 만지기 시작했고, 결국 1990년대 중반에 직장에서 프로그래밍을 배웠습니다. 2001년 무렵에는 괜찮은 펄(Perl) 해커가 되었고, 콜드 스프링 하버 연구소(CSHL)에서 여러 펄 모듈과 책의 저자인 링컨 스타인(Lincoln Stein) 박사의 랩에서 웹 개발자로 일하게 되었습니다. 그와 제 상사인 도린 웨어(Doreen Ware) 박사는 제가 그들이 원하는 프로그램을 이해할 수 있도록 인내심을 가지고 생물학을 가르쳐 주었습니다. 저는 Gramene.org라는 비교 식물 유전체학 데이터베이스에서 13년 동안 일하며 상당한 양의 과학을 배웠고, 동시에 프로그래밍 언어와 컴퓨터 과학을 계속 탐구했습니다.

링컨은 데이터와 코드부터 교육에 이르기까지 모든 것을 공유하는 데 열정적이었습니다. 그는 CSHL에서 유닉스 명령줄, 펄 프로그래밍, 생명정보학 기술을 가르치는 2주간의 집중 단기 코스인 ‘생물학을 위한 프로그래밍(Programming for Biology)’ 과정을 시작했습니다. 이 과정은 요즘은 파이썬을 사용하여 여전히 교육되고 있으며, 저도 여러 번 조교로 참여할 기회가 있었습니다. 저는 누군가가 자신의 연구를 발전시키기 위해 사용할 기술을 배우도록 돕는 것이 항상 보람차다는 것을 알게 되었습니다.

제가 CSHL에 재직하는 동안 보니 허위츠(Bonnie Hurwitz)를 만났고, 그녀는 결국 애리조나 대학교(UA)에서 박사 학위를 받기 위해 떠났습니다. 그녀가 UA에 자신의 연구실을 열었을 때 제가 첫 번째 직원이었습니다. 보니와 저는 수십 년 동안 함께 일했고, 가르치는 일은 제가 가장 좋아하는 업무 중 하나가 되었습니다. 링컨의 코스와 마찬가지로, 우리는 더 계산적인 접근 방식으로 분야를 넓히고 싶어 하는 과학자들에게 기본적인 프로그래밍 기술을 소개했습니다.

이 수업들을 위해 작성했던 자료 중 일부가 제 첫 번째 책인 Tiny Python Projects의 기초가 되었습니다. 그 책에서 저는 파이썬 언어 문법의 핵심 요소뿐만 아니라 프로그램이 올바르고 재현 가능(reproducible)한지 확인하기 위해 테스트를 사용하는 방법을 가르치려 노력했습니다. 이는 과학적 프로그래밍에 있어 매우 중요한 요소입니다. 이 책은 거기서부터 시작하여 생물학을 위한 프로그램을 작성하는 데 도움이 될 파이썬의 요소들에 집중합니다.

여기에는 프로그램이 올바르게 작동하는지 확인하는 데 사용할 수 있는 테스트 및 입력 데이터와 함께 여러 solution*.py 프로그램이 있습니다. 프로그램이 어떻게 작동해야 하는지 감을 잡기 위해 먼저 첫 번째 솔루션을 dna.py라는 프로그램으로 복사해 보겠습니다:

이제 인자 없이 프로그램을 실행하거나 -h 또는 --help 플래그를 사용하여 실행해 보세요. 사용법 문서가 출력될 것입니다(출력의 첫 단어가 usage임에 유의하세요):

$ ./dna.py
usage: dna.py [-h] DNA
dna.py: error: the following arguments are required: DNA

만약 “permission denied(권한 거부)”와 같은 에러가 발생한다면, chmod +x dna.py 명령을 실행하여 실행 비트를 추가함으로써 프로그램의 모드를 변경해야 할 수도 있습니다.

이것이 재현성(reproducibility)의 첫 번째 요소 중 하나입니다. 프로그램은 작동 방식에 대한 문서를 제공해야 합니다. 프로그램을 설명하기 위해 README 파일이나 논문을 두는 것이 일반적이지만, 프로그램 자체가 자신의 파라미터와 출력에 대한 문서를 제공해야 합니다. argparse 모듈을 사용하여 인자를 정의하고 검증할 뿐만 아니라 문서를 생성하는 방법을 보여드릴 것입니다. 이는 프로그램에 의해 생성된 사용법 문구가 틀릴 가능성이 없음을 의미합니다. README 파일이나 변경 로그 등이 프로그램의 개발 속도를 따라가지 못해 서로 맞지 않게 되는 경우와 비교해 보면, 이러한 방식의 문서화가 매우 효과적이라는 점을 이해하실 수 있을 것입니다.

사용법 라인을 보면 프로그램이 DNA와 같은 것을 인자로 기대한다는 것을 알 수 있으므로, 서열을 하나 입력해 보겠습니다. 로잘린드 페이지에 설명된 대로, 프로그램은 각 염기 A, C, G, T의 개수를 그 순서대로 출력하며, 각 숫자 사이에는 공백 하나가 들어갑니다:

$ ./dna.py ACCGGGTTTT
1 2 3 4

Rosalind.info 웹사이트에서 챌린지를 해결할 때 프로그램의 입력은 다운로드한 파일로 제공됩니다. 따라서 파일의 내용도 읽을 수 있도록 프로그램을 작성하겠습니다. cat(concatenate의 줄임말) 명령을 사용하여 tests/inputs 디렉토리에 있는 파일 중 하나의 내용을 출력할 수 있습니다:

$ cat tests/Input2.txt
AGCTTTTCATTCTGACTGCAACGGGCAATATGTCTCTGTGTGGATTAAAAAAAGAGTGTCTGATAGCAGC

이것은 웹사이트의 예제에 나와 있는 것과 동일한 서열입니다. 따라서 프로그램의 출력은 다음과 같아야 한다는 것을 알 수 있습니다:

$ ./dna.py tests/Input2.txt
20 12 17 21

소프트웨어를 테스트하는 핵심 요소는 알려진 입력을 사용하여 프로그램을 실행하고 그것이 올바른 출력을 생성하는지 확인하는 것입니다. 당연한 아이디어처럼 보일 수 있지만, 저는 단순히 프로그램을 실행만 할 뿐 올바르게 작동하는지 확인하지 않는 “테스트” 방식에 반대해 왔습니다.

new.py를 사용하여 프로그램 만들기 (Creating the Program Using new.py)

앞 섹션에서 보여드린 것처럼 솔루션 중 하나를 복사했다면, 처음부터 시작할 수 있도록 해당 프로그램을 삭제하세요:

$ rm dna.py

제 솔루션을 아직 보지 말고, 이 문제를 스스로 해결해 보시기 바랍니다. 필요한 정보가 모두 있다고 생각되면 바로 dna.py 버전을 작성하고 pytest를 사용하여 제공된 테스트를 실행해 보세요. 저와 함께 단계별로 프로그램을 작성하고 테스트를 실행하는 법을 배우고 싶다면 계속 읽어주세요.

이 책의 모든 프로그램은 하나 이상의 명령줄 인자를 받아들여 명령줄의 텍스트나 새 파일과 같은 출력을 생성합니다. 저는 항상 서문에서 설명한 new.py 프로그램을 사용하여 시작하지만, 이것이 필수는 아닙니다. 어떤 지점에서 시작하든 여러분이 원하는 방식으로 프로그램을 작성할 수 있지만, 여러분의 프로그램은 사용법 문구 생성 및 인자 유효성 검사와 같은 동일한 기능을 갖추어야 합니다.

01_dna 디렉토리에 dna.py 프로그램을 생성하세요. 이 디렉토리에 프로그램용 테스트 파일이 들어 있기 때문입니다. 저는 다음과 같이 dna.py 프로그램을 시작하겠습니다. --purpose 인자는 프로그램의 문서화에 사용됩니다:

$ new.py --purpose 'Tetranucleotide frequency' dna.py
Done, see new script "dna.py".

새로 만든 dna.py 프로그램을 실행하면 명령줄 프로그램에서 흔히 사용하는 다양한 유형의 인자들이 정의되어 있음을 알 수 있습니다:

$ ./dna.py --help
usage: dna.py [-h] [-a str] [-i int] [-f FILE] [-o] str

Tetranucleotide frequency

positional arguments:
  str                   A positional argument

optional arguments:
  -h, --help            show this help message and exit
  -a str, --arg str     A named string argument (default: )
  -i int, --int int     A named integer argument (default: 0)
  -f FILE, --file FILE  A readable file (default: None)
  -o, --on              A boolean flag (default: False)
  1. new.py에서 전달한 --purpose가 프로그램을 설명하는 데 사용되었습니다.
  2. 프로그램은 단일 위치 문자열 인자(positional string argument)를 받아들입니다.
  3. -h와 --help 플래그는 argparse에 의해 자동으로 추가되며 도움말 출력을 트리거합니다.
  4. 이는 문자열 값을 위한 짧은 이름(-a)과 긴 이름(--arg)을 가진 명명된 옵션(named option)입니다.
  5. 이는 정수 값을 위한 짧은 이름(-i)과 긴 이름(--int)을 가진 명명된 옵션입니다.
  6. 이는 파일 인자를 위한 짧은 이름(-f)과 긴 이름(--file)을 가진 명명된 옵션입니다.
  7. 이는 -o 또는 --on이 있으면 True, 없으면 False가 되는 불리언(Boolean) 플래그입니다.

이 프로그램에는 str 위치 인자만 필요하며, metavar 값으로 DNA를 사용하여 사용자에게 인자의 의미를 알려줄 수 있습니다. 다른 모든 파라미터는 삭제하세요. -h와 --help 플래그는 argparse가 내부적으로 사용하므로 직접 정의할 필요가 없습니다. 프로그램을 수정하여 다음과 같은 사용법이 출력되도록 만들어 보세요(아직 사용법을 출력하지 못하더라도 걱정하지 마세요, 다음 섹션에서 보여드리겠습니다):

$ ./dna.py -h
usage: dna.py [-h] DNA

Tetranucleotide frequency

positional arguments:
  DNA         Input DNA sequence

optional arguments:
  -h, --help  show this help message and exit

이것을 작동시킬 수 있다면, 이 프로그램이 정확히 하나의 위치 인자만을 받아들인다는 점을 말씀드리고 싶습니다. 만약 다른 개수의 인자로 실행하려고 하면 프로그램은 즉시 중단되고 에러 메시지를 출력할 것입니다:

$ ./dna.py AACC GGTT
usage: dna.py [-h] DNA
dna.py: error: unrecognized arguments: GGTT

마찬가지로 프로그램은 알 수 없는 플래그나 옵션도 거부할 것입니다. 매우 적은 양의 코드로 여러분은 인자를 검증하고 문서화된 프로그램을 구축했습니다. 이것은 재현성을 향한 매우 기본적이고 중요한 단계입니다.

argparse 사용하기 (Using argparse)

new.py에 의해 생성된 프로그램은 argparse 모듈을 사용하여 프로그램의 파라미터를 정의하고, 인자가 올바른지 검증하며, 사용자를 위한 사용법 문서를 생성합니다. argparse 모듈은 파이썬 표준 모듈이므로 항상 존재합니다. 다른 모듈들도 이러한 작업을 수행할 수 있으며, 프로그램의 이 부분을 처리하기 위해 원하는 어떤 방법이든 자유롭게 사용할 수 있습니다. 다만 여러분의 프로그램이 테스트를 통과할 수 있어야 합니다.

저는 Tiny Python Projects를 위해 new.py 버전을 작성했는데, 해당 책의 GitHub 저장소 bin 디렉토리에서 찾을 수 있습니다. 그 버전은 제가 여러분이 사용하기를 원하는 버전보다 다소 단순합니다. 먼저 이 초기 버전의 new.py를 사용하여 생성된 dna.py 버전을 보여드리는 것으로 시작하겠습니다:

#!/usr/bin/env python3
""" Tetranucleotide frequency """

import argparse

def get_args():
    """ Get command-line arguments """
    parser = argparse.ArgumentParser(
        description='Tetranucleotide frequency',
        formatter_class=argparse.ArgumentDefaultsHelpFormatter)
    
    parser.add_argument('dna', metavar='DNA', help='Input DNA sequence')
    
    return parser.parse_args()

def main():
    """ 여기서 작업을 수행합니다 """
    args = get_args()
    print(args.dna)

if __name__ == '__main__':
    main()
